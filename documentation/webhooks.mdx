Webhooks allow you to receive real-time HTTP notifications when events occur in the Sophic Engine platform. Instead of polling the API for changes, you register a URL and Sophic Engine sends a `POST` request to that URL whenever a relevant event is triggered.

## Setting up a webhook

Use the [Create webhook](/api-reference/webhooks/post-webhooks) endpoint to register a webhook URL. You can also manage webhooks through the dashboard.

Each webhook subscription requires:
- A destination **URL** (must be HTTPS)
- One or more **event types** to subscribe to

```bash
curl -X POST https://api.engine.usesophic.com/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://your-app.com/webhooks/sophic",
    "events": ["order.filled", "deposit.completed"]
  }'
```

## Webhook headers

Every webhook delivery includes the following HTTP headers:

| Header | Description |
|---|---|
| `Content-Type` | Always `application/json`. |
| `Webhook-Id` | The unique ID of the delivery (e.g. `whd_abc123`). Use this to deduplicate retried deliveries. |
| `Webhook-Timestamp` | Unix timestamp (seconds) of when the delivery was dispatched. Use this to reject stale messages. |
| `Webhook-Signature` | HMAC-SHA256 signature(s) of the payload. Multiple signatures are space-separated during secret rotation. |
| `Webhook-Attempt` | 1-indexed attempt number for this delivery. `1` on the first try, incremented on each retry. |
| `Event-Id` | The ID of the originating event (e.g. `evt_abc123`). Not present on test deliveries. |

## Receiving webhooks

When an event occurs, Sophic Engine sends a `POST` request to your registered URL with a JSON payload:

```json
{
  "id": "evt_abc123",
  "name": "order.filled",
  "resource": {
    "id": "ord_xyz789",
    "resource_type": "order"
  },
  "data": {
    // ... event-specific data
  },
  "created_at": "2026-01-15T10:30:00Z"
}
```

Your endpoint must return a `2xx` status code within **15 seconds** to acknowledge receipt. Any other status code, timeout, or connection error is treated as a failure. The maximum time Sophic Engine will wait for a response is 25 seconds (5s connect + 15s read + 5s write).

## Verifying webhook signatures

Every webhook request includes a signature in the `Webhook-Signature` header. **You must verify this signature** to confirm the request originated from Sophic Engine and was not tampered with.

The signature is computed using HMAC-SHA256 over the delivery ID, timestamp, and raw request body.

### Signature format

The `Webhook-Signature` header contains one or more signatures in the format:

```
v1,{hex_digest} v1,{hex_digest}
```

Multiple signatures are present during [secret rotation](#secret-rotation) — one for each active signing secret. A request is valid if **any** signature matches.

### Verification steps

<Steps>
  <Step title="Extract the headers">
    Read the `Webhook-Signature`, `Webhook-Id`, and `Webhook-Timestamp` headers from the incoming request.
  </Step>
  <Step title="Prepare the signed payload">
    Concatenate the timestamp, the delivery ID, and the raw request body, separated by periods (`.`):
    ```
    {webhook_timestamp}.{webhook_id}.{raw_body}
    ```
  </Step>
  <Step title="Compute the expected signature">
    Compute an HMAC-SHA256 using your webhook signing secret as the key and the prepared string as the message.
  </Step>
  <Step title="Compare signatures">
    Split the `Webhook-Signature` header on spaces. The request is valid if any of the provided signatures matches your computed signature. Always use a constant-time comparison function to prevent timing attacks.
  </Step>
</Steps>

### Example

```python
import hmac
import hashlib

def verify_webhook(payload: bytes, headers: dict, secret: str) -> bool:
    timestamp = headers["Webhook-Timestamp"]
    delivery_id = headers["Webhook-Id"]
    signatures = headers["Webhook-Signature"].split(" ")

    signed_payload = f"{timestamp}.{delivery_id}.".encode() + payload
    computed = hmac.new(
        secret.encode(),
        signed_payload,
        hashlib.sha256,
    ).hexdigest()
    expected = f"v1,{computed}"

    return any(hmac.compare_digest(expected, sig) for sig in signatures)
```

```javascript
const crypto = require("crypto");

function verifyWebhook(payload, headers, secret) {
  const timestamp = headers["webhook-timestamp"];
  const deliveryId = headers["webhook-id"];
  const signatures = headers["webhook-signature"].split(" ");

  const signedPayload = `${timestamp}.${deliveryId}.${payload}`;
  const computed =
    "v1," +
    crypto.createHmac("sha256", secret).update(signedPayload).digest("hex");

  return signatures.some(
    (sig) =>
      sig.length === computed.length &&
      crypto.timingSafeEqual(Buffer.from(computed), Buffer.from(sig))
  );
}
```

### Secret rotation

When you rotate your webhook signing secret, both the old and new secrets remain active during a transition period. During this window, Sophic Engine signs each delivery with all active secrets, producing multiple space-separated signatures in the `Webhook-Signature` header. This allows you to update your verification code without missing any deliveries.

## Rejecting stale webhooks

To protect against replay attacks, check the `Webhook-Timestamp` header. Reject any webhook where the timestamp is more than **5 minutes** from your server's current time.

```python
import time

def is_timestamp_valid(headers: dict, tolerance_seconds: int = 300) -> bool:
    timestamp = int(headers["Webhook-Timestamp"])
    return abs(time.time() - timestamp) < tolerance_seconds
```

<Warning>
Always validate the timestamp **before** processing the webhook. Replayed requests with old timestamps should be discarded even if the signature is valid.
</Warning>

## Webhook deliveries

A **Webhook Delivery** tracks the delivery attempts for an event to a webhook endpoint. When an event is dispatched, Sophic Engine creates a delivery for each matching webhook subscription. Each delivery has its own unique ID (the `Webhook-Id` header) and progresses through the following statuses:

| Status | Description |
|---|---|
| `pending` | The delivery has been created and is waiting to be dispatched. |
| `in_progress` | The delivery is actively being attempted. |
| `succeeded` | The endpoint returned a `2xx` response. |
| `failed` | All retry attempts have been exhausted without a successful response. |

### Retry schedule

Sophic Engine uses **at-least-once delivery**. If your endpoint does not return a `2xx` response, the delivery is retried with full jitter exponential backoff, up to **10 attempts**:

| Attempt | Average delay | Maximum delay |
|---|---|---|
| 1 | Immediate | — |
| 2 | ~0.5s | 1s |
| 3 | ~3s | 6s |
| 4 | ~18s | 36s |
| 5 | ~1m 48s | 3m 36s |
| 6 | ~10m 48s | 21m 36s |
| 7 | ~1h 4m | 2h 9m |
| 8 | ~6h 28m | 12h 57m |
| 9 | ~12h | 24h |
| 10 | ~12h | 24h |

The total delivery window spans approximately **43 hours on average** and up to **3.6 days** in the worst case. After 10 failed attempts, the delivery is marked as `failed` and no further retries are made.

### Delivery attempt errors

Each failed delivery attempt records the type of error encountered:

| Error type | Description |
|---|---|
| `http` | Your endpoint returned a non-2xx HTTP status code. |
| `timeout` | The connection or read timed out. |
| `dns` | DNS resolution for your endpoint failed. |
| `tls` | TLS certificate verification failed. |
| `connection` | Connection was refused, reset, or a network error occurred. |
| `validation` | The webhook URL failed validation (e.g. resolves to a private IP). |
| `unknown` | An unhandled exception occurred during delivery. |

## Handling duplicates

Because delivery is at-least-once, your webhook handler must be **idempotent**. Use the `Webhook-Id` header to deduplicate deliveries:

1. When you receive a delivery, check if you have already processed a delivery with that `Webhook-Id`.
2. If you have, return `200` immediately without reprocessing.
3. If you haven't, process the event and record the ID.

```python
def handle_webhook(request):
    delivery_id = request.headers["Webhook-Id"]

    if already_processed(delivery_id):
        return 200

    process_event(request.json())
    mark_processed(delivery_id)
    return 200
```

## Ordering

Events are delivered in the order they are created, but due to retries and network conditions, they may arrive out of order. Use the `created_at` timestamp on each event to determine the correct sequence. Do not rely on delivery order for business logic.

## Best practices

<CardGroup cols={2}>
  <Card title="Respond quickly" icon="bolt">
    Return a `2xx` response as fast as possible. Offload processing to a background job or queue rather than doing heavy work inline.
  </Card>
  <Card title="Use HTTPS" icon="lock">
    Your webhook endpoint must use HTTPS. Sophic Engine will not deliver events to insecure HTTP URLs.
  </Card>
  <Card title="Verify signatures" icon="shield-check">
    Always verify the `Webhook-Signature` header and reject requests with invalid or stale timestamps.
  </Card>
  <Card title="Handle retries gracefully" icon="rotate">
    Build your handler to be idempotent using `Webhook-Id`. The same event may be delivered multiple times.
  </Card>
  <Card title="Monitor failures" icon="chart-line">
    Set up alerting for webhook delivery failures. Use the [List events](/api-reference/events/get-events) endpoint to reconcile missed events.
  </Card>
  <Card title="Check the attempt number" icon="arrow-rotate-right">
    Use the `Webhook-Attempt` header to log and monitor how often retries are happening — high retry rates may indicate endpoint issues.
  </Card>
</CardGroup>

Use the [Test webhook](/api-reference/webhooks/post-webhooks-webhook_id-test) endpoint to send test deliveries to your URL during development. Test deliveries use the event name `test.event` and do not include the `Event-Id` header.
</Tip>
